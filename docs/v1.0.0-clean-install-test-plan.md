# Cidre v1.0.0 Clean-Install Test Plan

v0.13.0 improves the macOS-side bootstrap and adds Linux-side seed import and resume, but v1.0.0 remains the first milestone that should target a near-Ubuntu-level end-to-end install experience.

This document details the actionable test steps required for real Apple Silicon Mac validation of the v1.0.0 release.

Additional v1.0.0 requirements:

- Cidre image can be selected or installed from the macOS-side flow.
- firstboot OOBE starts without requiring the user to know `root/root`.
- firstboot creates or verifies the normal user.
- firstboot disables the temporary bootstrap mechanism after completion.
- `niri-cidre` desktop can be reached after firstboot.
- Cidre image artifact is generated reproducibly.
- Cidre image includes firstboot-root service.
- Cidre image includes seed/resume tools.
- Cidre image first boot is tested on Apple Silicon hardware.
- Cidre image root partition grows as expected.
- Cidre image root UUID behavior is verified.
- user is not required to know `root/root`.
- firstboot OOBE appears on first boot.
- user creation succeeds.
- seed/resume handoff succeeds.
- firstboot completion marker prevents repeated setup.
- temporary bootstrap mechanism is disabled after completion.

## Test Environment Prerequisites
* Target: Apple Silicon Mac (M1/M2/M3).
* Base System: Freshly installed ALARM (Arch Linux ARM) or Asahi Linux Minimal environment.
* Connection: Network interface (Wi-Fi or Ethernet) active.

## Execution Matrix Checklist

### Phase 0: Pre-installation Base Setup (Root Phase)
- [ ] Connect network interface and verify ping reachability.
- [ ] Install git using `pacman -Sy --needed git`.
- [ ] Clone the repository and run `./preinstall --check`. Verify dependencies and network checks report OK.
- [ ] Run `./preinstall --prepare` to configure sudo/wheel access and create a normal user.
- [ ] Verify that `%wheel ALL=(ALL:ALL) ALL` is configured under `/etc/sudoers.d/cidre-wheel`.

### Phase 1: Installation & Bootstrap
- [ ] Switch to the normal user with `su - <username>`.
- [ ] Run `./install --check` and verify command diagnostics report OK.
- [ ] Run `./install --dry-run --profile desktop` and confirm correct simulation.
- [ ] Run `./install --profile desktop`. Confirm successful pacman dependencies compilation.
- [ ] Verify logs are written to `~/.local/state/cidre/install.log`.

### Phase 2: Session Launch & Welcome
- [ ] Reboot system. Select the `Cidre` composition session at the `greetd` login greeter.
- [ ] Log in. Confirm successful compositor launch.
- [ ] Verify the visual TUI dashboard `cidre-welcome` executes upon session start.
- [ ] Run `cidre-firstboot --status` to check execution audit status.

### Phase 3: Hardware & Desktop Utilities
- [ ] Ghostty terminal opens via `Super + Enter`.
- [ ] fuzzel menu launcher opens via `Super + D`.
- [ ] Waybar status bar loads correct panel widgets (battery, audio indicators).
- [ ] Press `Print` and select area selection screenshots. Verify screenshots are saved under `~/Pictures/Screenshots/`.
- [ ] Toggle brightness hotkeys. Verify display levels adjust.

### Phase 4: Audio & Buffer Verification
- [ ] Connect headphones / speakers. Run `cidre-audio status`.
- [ ] Run audio tests. Verify pop pops popping sounds are resolved by setting buffer to stable.

### Phase 5: Updates, Snapshots, and Recovery
- [ ] Run `cidre-snapshot create --label "post-install-test"`.
- [ ] Make edits inside `~/.config/niri/config.kdl`. Run `cidre-maintenance drift` to detect differences.
- [ ] Run `cidre-update --check` to sync packages.
- [ ] Run `cidre-recovery restore latest` to roll back custom configuration changes.

## Boot validation requirements (v0.17.0)

The following must be verified before v1.0.0:

- Rootfs image inspected before boot (`cidre-rootfs-inspect` passes)
- Firstboot service enabled in image (symlink in `multi-user.target.wants`)
- Firstboot OOBE appears on first boot console
- User can proceed without knowing `root/root`
- Firstboot completion marker prevents repeated OOBE
- Boot logs can be collected with `cidre-boot-log-collect`
- Failure recovery path is documented

## Builder integration requirements (v0.18.0)

The following must be verified before v1.0.0:

- Builder configuration loads and directory presence checks pass (`cidre-builder-config`)
- Staging tree layout overlays sync with correct permission bounds (`cidre-builder-integrate`)
- Image build wrappers execute clean simulation paths (`cidre-builder-invoke`)
- Error scanning parse target warnings correctly (`cidre-builder-log`)
- Produced image registration writes to manifest schema version `0.18.0` (`cidre-builder-artifacts`)
- Promotion verification requires clean boot validation checks (`cidre-image-promote`)

## Real image build requirements (v0.19.0)

The following must be verified before v1.0.0:

- Host environment diagnostic check successfully detects git, tar, zstd (`cidre-build-environment`)
- High-level orchestrator plans dry-run builds cleanly (`cidre-real-image-build --dry-run`)
- Executing builder captures exit status and logs (`cidre-real-image-build --run`)
- Artifact registration updates manifest configurations (`cidre-image-register`)
- Registered files pass checksum integrity check (`cidre-image-verify`)
- Failure reports capture diagnostic parameters upon error (`cidre-build-failure-report`)
- Build report summaries write successfully (`cidre-image-build-report`)
## Controlled boot test requirements (v0.20.0)

The following must be verified before v1.0.0:

- Boot preflight successfully verifies host resources and local environment configuration (`cidre-boot-preflight`)
- Orchestrator handles boot testing states and writes clean boot results (`cidre-controlled-boot-test`)
- Human observations are recorded during testing (`cidre-boot-observe`)
- Failure classifications categorize boot states accurately (`cidre-boot-failure-classify`)

## Firstboot stability requirements (v0.21.0)

The following must be verified before v1.0.0:

- Firstboot service starts automatically on first boot
- OOBE output is clearly visible in the console
- Handoff file is generated and visible
- Failed firstboot runs can be diagnosed accurately (`cidre-firstboot-diagnose`)
- Firstboot retry path resets state markers safely (`cidre-firstboot-retry`)
- Completion markers prevent repeated firstboot OOBE executions
- Firstboot report compiles successfully into markdown format (`cidre-firstboot-report`)

## User phase requirements (v0.22.0)

The following must be verified before v1.0.0:

- Selected user exists inside home directory lists
- Selected profile parameter is valid against profiles indices
- Machine-readable handoff environment and JSON state files are generated correctly
- Normal user can safely import handoff parameters (`cidre-user-handoff`)
- Installer resumption preflight checks pass cleanly (`cidre-user-phase-verify`)
- User phase setup records started, completed, and failed execution state markers (`cidre-user-phase-state`)
- User phase execution report compiles successfully into markdown format upon error (`cidre-user-phase-report`)
- Recovery subcommands can display user phase setup details (`cidre-recovery`)

## Exit path requirements (v0.23.0)

The following must be verified before v1.0.0:

- Cidre state can be exported
- partition layout can be audited read-only
- uninstall plan can be generated
- macOS restore guide can be generated
- uninstall risk can be classified
- destructive disk operations require explicit confirmation
- macOS-side restore assistant exists
- user can return to macOS without guessing

## macOS restore requirements (v0.24.0)

The following must be verified before v1.0.0:

- macOS-side restore check passes
- `diskutil` layout can be collected
- startup disk state can be reviewed
- partition candidates can be reported as advisory only
- uninstall guide can be generated
- restore report can be generated
- no destructive command runs without explicit future guided mode

## Recovery UX requirements (v0.25.0)

The following must be verified before v1.0.0:

- normal desktop failure shows a clear recovery path
- safe mode shell can be started
- recovery screen can list actions
- doctor, recovery, state-export, and exit-plan can be run from recovery mode
- repeated desktop failure triggers safe mode recommendation
- failure report can be generated
- user is told when Rescue Slot or macOS Restore Assistant is required

## Rescue requirements (v0.26.0)

The following must be verified before v1.0.0:

- Rescue Slot plan can be generated
- Rescue profile can be validated
- Rescue rootfs artifact can be built and inspected
- Rescue tools can scan main Cidre candidates
- Rescue mount defaults to read-only
- Rescue can export main Cidre state and logs
- Rescue can inspect main kernel, initramfs, and boot files
- Recovery Screen points to Rescue Slot when main Cidre cannot boot

## Rescue boot integration requirements (v0.27.0)

The following must be verified before v1.0.0:

- rescue boot plan can be generated
- rescue slot metadata can be generated and validated
- macOS-side rescue disk audit can be generated
- rescue create dry-run shows exact intended steps
- no destructive operation runs by default
- real rescue boot validation checklist exists
- rescue boot status is visible from doctor and recovery
